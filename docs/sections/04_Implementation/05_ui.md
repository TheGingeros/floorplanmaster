# 4.5 Uživatelské rozhraní

Uživatelské rozhraní addonu sdružuje tři nezávislé, ale vzájemně koordinující se mechanismy: N-panel pro přístup k parametrům a nástrojům, overlay manager jako centrální dispatcher GPU vizualizace ve viewportu a sdílený stav výběru jako komunikační bod mezi operátory a UI.

## N-panel

N-panel je záložka „FloorPlanMaster" v postranní liště 3D viewportu a tvoří hlavní centrum interakce s addonem. Je strukturován do čtyř sekcí odpovídajících různým kontextům práce.

Sekce **Vlastnosti stěny** je viditelná pouze při vybrané stěně. Zobrazuje tloušťku a výšku stěny jako editovatelná pole s live aktualizací a seznam otvorů s možností přidání nového nebo odstranění existujícího. Změny parametrů se okamžitě projeví ve viewportu díky update callbackům, jež spouštějí synchronizaci Vrstvy 3.

Sekce **Nástroje** obsahuje tlačítka pro spuštění dostupných nástrojů — tužkový nástroj (FP1) a vložení místnosti (FP2). Je vždy viditelná bez ohledu na aktuální výběr.

Sekce **Místnosti** zobrazuje seznam všech detekovaných místností s jejich jmény a plochou. Každou místnost lze rozbalit pro zobrazení dalších detailů a kliknutím ji aktivovat — sdílený stav výběru se nastaví na danou místnost a lokace kamery se přesune k jejímu centroidu. Jméno každé místnosti je editovatelné přímo v seznamu.

Sekce **Nastavení** umožňuje upravit výchozí tloušťku a výšku stěny pro nové kresby; tyto hodnoty jsou součástí `.blend` souboru a jsou specifické pro každý projekt.

Synchronizace jmen místností mezi N-panelem a grafem místností probíhá přes vlastnosti Blender objektu, které jsou serializovatelné do `.blend` souboru. Při zobrazení panelu jsou jména načtena z objektu; při editaci jsou změny okamžitě zapsány zpět a perzistovány.

## Overlay manager

Ve standardním Blenderu addon, který potřebuje kreslit do viewportu, musí registrovat vlastní draw handler skrze Blender API. Pokud by takový handler registroval každý modul zvlášť a nezávisle, bylo by obtížné zaručit pořadí vykreslování, správně spravovat jejich životní cyklus a zabránit zbytečným voláním při neaktivním addonu.

FloorPlanMaster tento problém řeší centralizovaným overlay managerem. Celý addon registruje pouze jeden pár draw handlerů — jeden pro 3D world-space geometrii a jeden pro 2D screen-space prvky. Každý handler pouze sekvenčně volá seznam registrovaných callback funkcí se současným kontextem.

Jednotlivé funkce addonu přidávají a odebírají své kreslicí callbacky z těchto centrálních seznamů. Persistentní vrstvy, jako je zvýraznění vybrané stěny nebo místnosti, jsou registrovány jednou při aktivaci addonu a odregistrovány při jeho deaktivaci. Tranzientní vrstvy, například kreslicí náhled tužkového nástroje, jsou registrovány při spuštění příslušného operátoru a odregistrovány při jeho ukončení. Díky tomuto přístupu je pořadí vykreslování konzistentní, přidání nové overlay vrstvy vyžaduje pouze implementaci jedné funkce a uvolnění zdrojů při deaktivaci addonu proběhne správně pro všechny vrstvy najednou.

## Sdílený stav výběru

Sdílený stav výběru je modul udržující jedinou instanci s informací o aktuálně vybrané stěně nebo místnosti — konkrétně jejich UUID. Protože modální operátory, N-panel, overlay vrstvy a selection operátor jsou v Blenderu samostatné Python objekty bez přímé reference na sebe, sdílený stav slouží jako komunikační bod: výběr se nastaví z raycasting operátoru a N-panel, overlay vrstvy i zvýraznění výběru jej čtou při každém překreslení.

Toto řešení bylo zvoleno jako alternativa k ukládání stavu přes Blender vlastnosti na scéně, což by vyžadovalo serializaci UUID jako textový řetězec, update callbacks a přidanou komplexitu. Sdílený stav výběru je čistě runtime informace, která není nutná pro persistenci do `.blend` souboru, a proto je udržována přímo jako modulová proměnná.
